# Data Analytics Workflow: Implementation Plan

## Executive Summary

This plan builds on the existing Metabase + PostgreSQL + dlt infrastructure to add a transformation layer (dbt) and orchestration (Prefect) for marketing/e-commerce analytics.

**Current State**: Metabase v0.59.5 + PostgreSQL 17 + dlt pipelines  
**Target State**: Add dbt transformations + Prefect orchestration + data quality  
**Timeline**: 3-4 weeks for Phase 1

---

## Interpretation of Intent

From the brainstorm document, the key priorities are:
1. **Research first**: Identify common connector failures before building
2. **Standards-based**: Use existing dbt/Prefect/OpenLineage patterns
3. **Docker-first**: All components must run cleanly in existing Docker Compose
4. **Worktree approach**: Parallel subagents per concern
5. **Minimal Phase 1**: One ingestion path, one dbt path, one dashboard
6. **Anti-overengineering**: Don't build Phase 2 during Phase 1

## Research Completed ✓

### 1. dlt Connector Failure Modes
**Status**: Documented in `study/integrations/dlt-connector-failure-modes.md`

| Connector | Key Risks | Mitigation |
|-----------|-----------|------------|
| **GA4** | 200K tokens/day quota, sampling at 10M rows, concurrent request limits | Exponential backoff, date chunking, rowLimit parameter |
| **Facebook Ads** | 200 calls/hour rate limit, token expiration (190/463 errors), app review | Token refresh logic, rate limit headers, app review planning |
| **PrestaShop** | 401 auth failures, 500 server errors, SSL cert issues, timeouts | Retry logic, connection pooling, certificate validation |

### 2. dbt + PostgreSQL Compatibility
**Status**: Research complete - dbt-postgres 1.8+ fully supports PostgreSQL 17

- **Adapter**: `dbt-postgres>=1.8.0` compatible with PostgreSQL 17
- **Performance**: Use `unlogged: true` for staging tables, add indexes on marts
- **Permissions**: dbt user needs `CREATE SCHEMA` on target database
- **Incremental**: Use `delete+insert` strategy (default) for PostgreSQL

### 3. Prefect Docker Integration
**Status**: Research complete - Prefect 3.x Server + Worker architecture fits existing stack

- **Resource Limits**: Match existing pattern (2GB memory, 2 CPU per container)
- **Network**: Join existing `metabase-net` for connectivity to PostgreSQL
- **Storage**: PostgreSQL for Prefect state (can share metabase-db or separate)
- **Worker**: Docker work pool for isolated flow execution

### 4. Data Quality Edge Cases
**Status**: Documented - dbt-expectations covers common marketing data issues

| Issue | Test | Priority |
|-------|------|----------|
| Negative revenue | `expect_column_values_to_be_between` with `min_value: 0` | High |
| Duplicate orders | `expect_column_values_to_be_unique` on order_id | High |
| Timezone mismatch | Set `dbt_date:time_zone: 'Europe/Prague'` | Medium |
| NULL customer_ids | `expect_column_values_to_not_be_null` where applicable | Medium |
| Stale data | `expect_row_values_to_have_recent_data` | High |

### Docker Compose Extension
```yaml
# Add to existing docker-compose.yml
services:
  # ... existing metabase and metabase-db ...
  
  dbt:
    image: ghcr.io/dbt-labs/dbt-postgres:1.9.latest
    container_name: dbt
    volumes:
      - ./dbt:/usr/app/dbt
      - ./dbt/profiles.yml:/root/.dbt/profiles.yml
    working_dir: /usr/app/dbt
    environment:
      DBT_PROFILES_DIR: /root/.dbt
    networks:
      - metabase-net
    profiles: ["dbt"]  # Only run when explicitly requested
    
  prefect-server:
    image: prefecthq/prefect:3-latest
    container_name: prefect-server
    ports:
      - "127.0.0.1:4200:4200"
    environment:
      PREFECT_HOME: /data
      PREFECT_API_URL: http://localhost:4200/api
    volumes:
      - prefect-data:/data
    command: prefect server start --host 0.0.0.0
    networks:
      - metabase-net
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:4200/api/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      
  prefect-worker:
    image: prefecthq/prefect:3-latest
    container_name: prefect-worker
    environment:
      PREFECT_API_URL: http://prefect-server:4200/api
    command: prefect worker start --pool docker-pool --type docker
    networks:
      - metabase-net
    depends_on:
      - prefect-server
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock

volumes:
  metabase-db-data:  # existing
  prefect-data:      # new
```

3. **Do not** add Great Expectations yet - dbt tests first
4. **Do not** add OpenLineage yet - lineage after stability
5. **Do not** build custom connectors - use dlt verified sources
6. **Do not** create generic frameworks - solve concrete problems
7. **Do not** optimize for scale - optimize for first workload
8. **Do not** add Phase 2 features during Phase 1
9. **Do not** create custom MCP tools - use existing ones

## Integrated Plan: Work Item Details

### 1. e2e-verification: End-to-End Pipeline Test

#### Objective
Validate the complete marketing analytics pipeline using real API credentials in an isolated test environment, ensuring data flows correctly from source APIs through dlt → dbt → PostgreSQL.

#### Scope
- **In Scope**: GA4, GAds, FB Ads, PrestaShop source validation; dlt sync verification; dbt run/test; data quality checks
- **Out of Scope**: Performance/load testing, penetration testing, cross-client data isolation validation

#### Constraints
| Type | Constraint | Impact |
|------|------------|--------|
| Technical | API rate limits (GA4: 1,200/day, FB: 200/hour) | Tests must implement backoff delays |
| Business | Real credentials expose production client data | Requires legal/compliance review |
| Resource | Need complete credential set + SOPS setup | 3-5 day lead time for Google approval |

#### Risks & Mitigation
| Risk | Severity | Mitigation |
|------|----------|------------|
| Credential exposure in logs | Critical | Sanitize all output; use `grep -v` filters; regex redaction |
| API rate limit exhaustion | High | Test-specific backoff; single-source mode (`--source`) |
| Production data contamination | High | Use `client_id="test"` with isolated datasets |
| Credential rotation post-test | Medium | Automated 90-day rotation in SOPS metadata |

#### Data Requirements
| Source | Volume | Freshness | Dataset |
|--------|--------|-----------|---------|
| GA4 | 10K-1M rows | <24h | `ga4_test.traffic` |
| GAds | 5K-100K rows | <24h | `gads_test.campaigns` |
| FB Ads | 10K-500K rows | <24h | `fbads_test.ads` |
| PrestaShop | 1K-50K rows | <1h | `prestashop_test.orders` |

#### Credential Requirements
| Source | Credentials | Storage | Rotation |
|--------|-------------|---------|----------|
| GA4 | PROPERTY_ID, CLIENT_ID, CLIENT_SECRET, REFRESH_TOKEN | SOPS | 6 months |
| GAds | DEVELOPER_TOKEN, CLIENT_ID, CLIENT_SECRET, REFRESH_TOKEN, CUSTOMER_ID | SOPS | 6 months |
| FB Ads | AD_ACCOUNT_ID, ACCESS_TOKEN | SOPS | 60 days |
| PrestaShop | SHOP_URL, API_KEY | SOPS | Quarterly |

#### Acceptance Criteria
```
AC1: Pipeline completes with all 4 sources (exit code 0)
AC2: Data lands in PostgreSQL (>0 rows per table)
AC3: dbt models materialize without errors
AC4: dbt tests pass (100% success rate)
AC5: Zero credential exposure in logs (verified via grep)
AC6: Test data isolated (no prod schema modifications)
```

#### Dependencies
| Dependency | Blocking | Lead Time |
|------------|----------|-----------|
| SOPS setup (`.sops.yaml`, Age keys) | Yes | 1 day |
| Test credentials procurement | Yes | 3-5 days |
| dbt mart model completion | Yes | 2 days |

#### Deliverables
| Artifact | Location | Purpose |
|----------|----------|---------|
| `e2e_test_harness.py` | `tests/e2e/` | Main test harness with CI/CD exit codes |
| `credentials_loader.py` | `tests/e2e/` | Secure loader (SOPS/Ansible Vault/env) |

## Security & Credential Management

### Secure Supply Chain
```
1. PROCUREMENT
   ├── GCP Console → OAuth 2.0 credentials
   ├── Meta Developers → Marketing API token
   ├── Google Ads API Center → Developer token
   └── PrestaShop → Webservice key (read-only)

2. ENCRYPTION
   ├── Immediate: sops --encrypt --in-place
   ├── Age public key in .sops.yaml
   └── Private key: ~/.config/sops/age/keys.txt (0600)

3. STORAGE
   ├── Repository: .sops/secrets/ (encrypted)
   ├── CI/CD: GitHub Secrets (masked)
   └── Production: Ansible Vault

4. RUNTIME INJECTION
   ├── export $(sops -d .sops/secrets/test.yaml | xargs)
   ├── Pipeline reads from env (never logs values)
   └── Subprocess inherits filtered environment

5. EXECUTION
   ├── Docker run with --env-file <(sops -d ...)
   ├── Logs filtered: grep -v -i "token|secret|password"
   └── Temp files: mktemp + delete on exit

6. ROTATION (post-test)
   ├── Rotate OAuth refresh tokens
   ├── Generate new PrestaShop API key
   └── Re-encrypt with SOPS
```

### Log Sanitization
```python
SANITIZE_PATTERNS = [
    (r'Bearer\s+\S+', 'Bearer [REDACTED]'),
    (r'token["\']?\s*[:=]\s*["\']?\S+', 'token=[REDACTED]'),
    (r'secret["\']?\s*[:=]\s*["\']?\S+', 'secret=[REDACTED]'),
]

def sanitize_log(text: str) -> str:
    for pattern, replacement in SANITIZE_PATTERNS:
        text = re.sub(pattern, replacement, text, flags=re.I)
    return text
```

With:
```yaml

    column_B: clicks
    or_equal: true
    row_condition: "clicks IS NOT NULL AND impressions IS NOT NULL"
```
